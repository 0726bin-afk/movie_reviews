## 0. 합의된 분류 기준

### 영향도 (Impact)
다른 모듈/팀원이 해당 파일을 import하거나 그 산출물에 의존하는 정도.
- **H (高)**: 3명 중 2명 이상이 의존, 또는 교체 시 재빌드·재임베딩·스키마 마이그레이션 필요
- **L (低)**: 해당 레이어 내부에서만 쓰임, 교체해도 파급 없음

### 복잡도 (Complexity)
구현 난이도 + 엣지케이스 + 외부 의존(비동기·상태).
- **H (高)**: 상태 머신·동시성·재시도·스왑 절차·프롬프트 튜닝 루프
- **L (低)**: 단순 CRUD, 설정, 텍스트 상수, 얇은 래퍼

### 표기
`영향도/복잡도` = `H/H`, `H/L`, `L/H`, `L/L` 네 단계.
🚧 = **선결 블로커** (이 파일이 없으면 다른 팀원이 대기).

기획안 §6에서 중복되거나 모호했던 구간을 다음과 같이 확정:

| 구간 | 경계 |
| --- | --- |
| **pgvector 스키마** | `db/schema.sql` owner는 **최정빈** (전체 스키마 관점). 안도겸은 `rag/retrievers/self_query.py`에서 **쿼리·인덱스 파라미터 튜닝만**. 스키마 변경은 PR로 요청. |
| **프롬프트** | **태깅 프롬프트(`ingestion/tagging/prompts.py`) = 이경욱**, **RAG 프롬프트(`rag/prompts/*`) = 안도겸**. |

## 5. 관찰 및 완화안

### 관찰 1 — 안도겸 워크로드 편중
전체 파일의 약 절반, H/H 파일 4개 중 3개가 안도겸. 기획안 §6을 그대로 매핑한 결과.

**완화안**
- 1주차: `providers/base.py`, `rag/state.py`를 **최우선 1~2일 내 고정** → 이경욱·최정빈이 병렬 진행 가능해짐.
- 2주차 LangGraph 리팩토링 단계에서 **`route_query.py`를 최정빈에게 이관** 고려 (프롬프트·분류는 상대적으로 독립적이고, 최정빈이 API 라우팅 설계와 맥락이 맞음).

### 관찰 2 — "미정" 7개 파일의 실질 owner 필요
3.5주 일정에서 "누가 먼저 막힘"이 큰 위험. 세 명 중 한 명을 **공통 기반 작업 오너로 암묵 지정** 권장.

**추천**: 안도겸. 이미 `state.py`·`base.py` 같은 블로커를 잡고 있어서 `core/types.py`·`config/settings.py`·`pyproject.toml`까지 묶어 **스캐폴딩 1일 몰아서 처리**하면 모두가 그날 저녁부터 병렬 진행 가능.

### 관찰 3 — 프론트와 백엔드 완료 시점 어긋남 주의
`frontend/app.py`(최정빈)는 `api/routes/chat.py` 완성 전까지 mock response로 개발해야 함. 1주차 말에 API 스펙(`api/schemas.py`) 먼저 fix → 이후 병렬 진행.
