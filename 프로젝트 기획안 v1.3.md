## 1. 프로젝트 개요

### 1.1. 프로젝트명
**리뷰를 활용한 영화 가이드 챗봇 서비스** (별칭: *지능형 영화 도슨트 앱*)

### 1.2. 한 줄 소개
> 흩어진 영화 리뷰와 TMI 정보를 LLM 기반 RAG로 통합하여, 사용자에게 **"관람 전·중·후"** 모든 순간의 궁금증을 해결해주는 대화형 영화 가이드 서비스.

### 1.3. 기획 배경 및 필요성

1. **리뷰의 파편화**
   - 영화 리뷰 데이터는 방대하지만, 사용자가 핵심을 파악하려면 수많은 텍스트를 직접 읽고 비교해야 함.
   - 동일 영화에 대한 상반된 평가가 공존해 객관적 판단이 어려움.

2. **흩어진 지식의 통합 필요**
   - 촬영지·비하인드·OST·옥에 티 등 TMI 정보가 블로그·커뮤니티·SNS에 산재.
   - 한 번에 확인할 수 있는 통합 채널의 부재.

3. **감상 경험의 확장**
   - "아는 만큼 보인다"는 전제 아래, 핵심 키워드와 숨겨진 정보를 연결하여 영화 감상 경험을 풍부하게 하는 **디지털 도슨트** 서비스를 지향.

## 2. 목표 및 범위

### 2.1. 프로젝트 목표
- **핵심 목표**: LLM(RAG) 기반으로 영화 리뷰·TMI를 통합 검색·분석하여 자연어 질의에 답변하는 챗봇 서비스 구축.
- **세부 목표**
  - 영화 리뷰 데이터를 **의미 기반(Semantic)**으로 검색 가능하게 구성.
  - 단순 평점을 넘어 **논리적 호불호 분석**과 **장르별 가변 키워드** 분석 제공.
  - TMI(촬영지, 캐스팅 비화, OST, 옥에 티 등)를 LLM 그라운딩으로 수집·DB화.
  - **의도적으로 난이도 높은 영화군(호불호 뚜렷·난해한 서사·풍부한 TMI)을 포함**시켜 RAG 응답 품질의 한계를 조기에 노출하고 개선 포인트를 확보.

### 2.2. 프로젝트 범위
| 구분 | 포함 (In-scope) | 제외 (Out-of-scope) |
| --- | --- | --- |
| 데이터 | 흥행작 중심 + 소수의 비흥행작을 **장르·특성별 균형**으로 선정한 후보 약 34편 (§2.4 참조) | 전체 영화 DB 구축, 실시간 개봉작 자동 갱신 |
| 기능 | RAG 기반 Q&A, 리뷰 요약, TMI 검색, 호불호 분석 | 개인화 추천(협업필터링), 소셜 기능, 결제 |
| 플랫폼 | 웹 서비스 (반응형) | 네이티브 모바일 앱 (시간 여유 시 Stretch) |
| 사용자 | 일반 사용자 (연령 제한 없음) | 특정 플랫폼 가입자 한정 |

### 2.3. 타겟 사용자(Persona)
- **Primary**: 영화 관람 전 정보를 찾아보는 일반 관객 (20~50대)
- **Secondary**: 영화 깊이 파는 시네필·영화 마니아 — TMI·비하인드 탐색용
- **공통 니즈**: "이 영화 볼만해?", "나와 맞는 영화야?", "이 장면의 비하인드는?"

### 2.4. 영화 선정 기준 및 후보군 (신규)

데이터셋은 **LLM 답변·추론 성능을 다면적으로 검증**할 수 있도록 장르별 균형과 특성별 분포를 동시에 고려해 구성한다. 흥행작만 모으면 유사한 리뷰·TMI 패턴에 치우쳐 RAG 품질의 한계가 드러나지 않기 때문에, **의도적으로 어려운 영화군**을 함께 포함시킨다.

#### 2.4.1. 선정 축 (3가지 특성)

| 축 | 의의 | 검증 대상 |
| --- | --- | --- |
| **① 호불호가 뚜렷한 영화** | 상반된 평점·리뷰가 공존 | 논리적 호불호 분석, 상반 리뷰 대조 요약 품질 |
| **② 서사·구조가 난해한 영화** | 해석 여지가 크고 몰입 장벽 존재 | 줄거리 요약 정확도, 상징·장면 해설 능력 |
| **③ TMI·비하인드가 풍부한 영화** | 촬영지·제작 일화·OST·문화 맥락이 다량 축적 | 그라운딩 품질, 출처 제시율, 멀티소스 통합 |

이에 더해 SF·액션·스릴러·드라마·애니메이션·히어로·판타지·실화 기반 등 **장르 편향을 최소화**하도록 배분한다.

#### 2.4.2. 후보 영화군 (총 34편)

| 분류 | 후보작 |
| --- | --- |
| ① 호불호 뚜렷 | 라라랜드, 조커, 그대들은 어떻게 살 것인가, 인어공주(실사), 이터널스, 비상선언, 리얼, 엘리멘탈, 썬더볼츠*, 매드맥스: 분노의 도로 |
| ② 난해한 서사·구조 | 인셉션, 인터스텔라, 메멘토, 매트릭스, 올드보이, 오펜하이머, 신세기 에반게리온 |
| ③ TMI·비하인드 풍부 | 기생충, 스파이더맨: 노웨이 홈, 대부, 살인의 추억, 듄, 아바타: 불과 재 |
| 장르 균형용 대표·흥행작 | 범죄도시 3, 범죄도시 4, 베테랑 2, 파묘, 서울의 봄, 인사이드 아웃 2, 스즈메의 문단속, F1, 왕과 사는 남자, 어쩔 수가 없다, 살목지 |

> 상기 후보군은 **초기 선정 리스트**이며, 실제 데이터 수집 단계에서 **크롤링 부하·데이터 품질·서비스 범위 제약** 등에 따라 일부는 최종에서 제외될 수 있다. 다수 작품이 복수 축에 해당하나(예: 기생충·오펜하이머는 TMI+난해 동시), 위 표는 **대표 축 중심 분류**이다.

## 3. 핵심 기능 정의

### 3.1. 지능형 리뷰 분석 시스템
| 기능 | 설명 |
| --- | --- |
| **OSS LLM 기반 태그 생성** | Watcha에서 크롤링한 리뷰 원문을 **Open LLM(Gemma4-2B)**에 입력하여 리뷰별 핵심 태그 자동 생성 → 메타데이터로 저장 |
| 핵심 키워드 연결 | 연기·연출·서사·고증 등 카테고리별 리뷰 분류 → 메타데이터 결합 |
| 장르별 가변 키워드 | SF/사극: '고증', '세계관 구축' / 액션·공포: '타격감', '점프 스케어', '긴장감' 강화 |
| 논리적 호불호 분석 | 5점 vs 1점 상반 리뷰를 대조 제공 → 사용자 성향별 판단 지원 |
| 민감 정보 판별 | '노출', '베드신', '쿠키' 등 키워드 추출 → 관람 편의 정보 안내 |

#### 3.1.1. 태그 정의 (19개 고정 태그)

리뷰 분류에 사용하는 태그는 아래 19개로 고정한다.

| 분류 | 태그 목록 |
| --- | --- |
| 전체 평가 | `전체긍정` / `전체부정` / `전체복합` |
| 연기 | `연기좋음` / `연기나쁨` |
| 연출 | `연출좋음` / `연출나쁨` |
| 서사 | `서사좋음` / `서사나쁨` |
| 비주얼 | `비주얼좋음` / `비주얼나쁨` |
| 음악 | `음악좋음` |
| 분위기 | `분위기가벼움` / `분위기무거움` |
| 고증 | `고증좋음` / `고증나쁨` |
| 기타 | `주의사항` / `TMI` / `장르특성` |

#### 3.1.2. 태그 생성 파이프라인
```
Watcha 리뷰 크롤링(Playwright) ─▶ 전처리(중복/스팸 제거)
                              ─▶ Open LLM (Gemma4-2B) 프롬프트 입력
                              ─▶ 리뷰별 태그 n개 추출 (JSON, 19개 고정 태그 중 선택)
                              ─▶ 통합 데이터셋 메타데이터 병합
                              ─▶ Supabase(pgvector) 임베딩 시 메타 컬럼·필터 조건으로 활용
```
- **모델 선택 사유**: 경량(2B) OSS로 로컬·무료 실행 가능, 한국어 리뷰 태깅에 충분한 성능, API 쿼터 의존 제거.
- **활용**: Self-Querying Retriever에서 태그를 필터 조건으로 활용 → 검색 정밀도 향상.
- **산출 예시**: `"연기가 정말 좋았다..." → ["전체긍정", "연기좋음", "감정몰입"]`

### 3.2. TMI 도슨트 기능
| 기능 | 설명 |
| --- | --- |
| 사전 수집 TMI | 촬영지·캐스팅 비화·OST·옥에 티를 카테고리화 |
| LLM 그라운딩 | **SerpAPI**로 최신 정보 수집 → 정제 후 DB 구조화 저장 |
| 실시간 Q&A | RAG + DB + 필요 시 추가 그라운딩 → 답변 생성 |
| 답변 결과 재저장 | 반복 질의에 대한 응답 속도·일관성 확보 |

### 3.3. 대화형 상담 세션
- 질문 유형 자동 라우팅 (영화 기본정보 / 리뷰 분석 / TMI / 호불호 / 추천)
- 답변 근거(출처 리뷰 원문, TMDB 링크) 함께 제시 → 신뢰도 확보

## 4. 시스템 아키텍처

### 4.1. 데이터 전략 — Supabase 단일화 (pgvector)

본 프로젝트는 **Supabase(PostgreSQL + pgvector 확장) 단일 DB** 로 Semantic·Keyword·Cache 레이어를 모두 수용한다. 당초 ChromaDB + Supabase 이원화 구조를 검토했으나 (1) MVP 스케일(약 2만 벡터 내외)에서 pgvector HNSW 인덱스의 성능 손실이 사실상 없고, (2) **의미 검색과 SQL JOIN을 단일 쿼리**로 결합할 수 있어 복합 조건(예: "봉준호 감독 4점 이상 리뷰 중 유사 문장")이 간결해지며, (3) 저장소 이원화에 따르는 동기화·배포 복잡도를 원천 제거할 수 있어 단일화로 최종 확정.

| 레이어 | 저장 위치 | 비고 |
| --- | --- | --- |
| Semantic | `reviews.embedding`, `tmi.embedding` (vector, HNSW 인덱스) | Self-Query Retriever 필터 대상 |
| Keyword/Relational | `movies`, `reviews`, `tmi` 일반 컬럼·FK | SQL 조건·조인 |
| Cache | `qa_cache` — `question_hash` B-tree + `question_embedding` HNSW | 정확 매칭 + 유사 매칭 이중 레이어 |
| Logs·Eval | `query_log`, `eval_set` | MVP 성공 기준 측정 |

**주요 설계 포인트**
- **HNSW 인덱스**: `USING hnsw (embedding vector_cosine_ops)` — 수십만 벡터까지 실질적 성능 이슈 없음.
- **ID 일원화**: 벡터와 원문 레코드가 동일 PK를 공유해 조인 단순화.
- **재임베딩 마이그레이션 절차**: 임시 컬럼 생성 → 배치 업데이트 → 컬럼 스왑(무중단) — 임베딩 모델 교체 시 표준 절차로 운영.

### 4.2. 기술 스택

| 계층 | 기술 | 비고 |
| --- | --- | --- |
| LLM (Primary) | **Google Gemini** | 메인 답변 생성 |
| LLM (Secondary) | **미정** | A/B 비교·폴백·빠른 분류 (후보: Claude / GPT-4o / Gemma 등) |
| RAG 오케스트레이션 | **LangGraph** | State 기반 상태 머신, 조건부 분기, 멀티턴 Checkpointer |
| RAG 컴포넌트 | **LangChain** | Self-Query Retriever, Prompt, Output Parser, LCEL Runnable |
| 통합 데이터 저장소 | **Supabase (PostgreSQL + pgvector, HNSW)** | Semantic + Keyword + Cache 단일화 |
| 백엔드 | **FastAPI** | LangChain/LangGraph Python 생태계 연계, 비동기 스트리밍, OpenAPI 자동 |
| 프론트엔드 | **Streamlit** (MVP) → **Next.js** (Stretch) | 빠른 프로토타이핑 후 정식 UI |
| 크롤러 | **Playwright 단독** | Watcha Pedia 동적 렌더링·브라우저 자동화 일원화 |
| 외부 API | TMDB API, **SerpAPI** | 영화 정보·실시간 그라운딩 |
| 임베딩 모델 | **Gemma4-2B** | 리뷰 태그 생성 및 임베딩 |
| 배포 | Supabase + Vercel or Render | 무료 티어 우선 |

> **RAG 개발 전략 — LangGraph + LangChain 혼합**
> ① 응답 흐름(캐시 체크 → 라우팅 → 검색 → 그라운딩 → 생성 → 캐시 저장)이 본질적으로 **상태 머신**이므로 LangGraph로 오케스트레이션.
> ② Self-Query Retriever·Prompt Template·LCEL Runnable 등 풍부한 컴포넌트는 LangChain을 노드 내부에서 그대로 활용.
> ③ Checkpointer를 Supabase Postgres에 연결해 멀티턴 상태·인프라를 단일 DB로 통합.

> **크롤러 — Playwright 단독화**
> requests·BeautifulSoup 이원화를 제거하고 Playwright로 통일. Watcha Pedia의 동적 렌더링·무한 스크롤 대응이 필요하며, User-Agent 제어·요청 간격 조절·스크린샷 기반 디버깅도 한 도구에서 처리 가능.

### 4.3. 서비스 흐름 (요약)
```
[데이터 수집]
 TMDB API ────────────────┐
 Watcha 크롤링(Playwright) ┼─▶ 통합 데이터셋(JSON) ─▶ 키워드·장르 태깅(Gemma4-2B)
 SerpAPI ──────────────────┘                     └─▶ Supabase (pgvector embedding + 원문·메타 통합)

[질의 응답]  *LangGraph State Machine*
 사용자 질문 ─▶ [cache_check: hash → embedding 유사도 순 조회]
           ─▶ [route_query: 5종 유형 분류]
           ─▶ [retrieve: Self-Query Retriever on pgvector + SQL JOIN]
           ─▶ (필요시) [ground_serpapi]
           ─▶ [generate: LangChain prompt | LLM] ─▶ 채팅 UI
           └─ [save_cache: question_hash·embedding 재저장]
```
